# Figma Slides 迁移方案

## 目标

把当前基于 React/Vite 的 web slides，迁移为一套以 Figma Slides 为主稿源的演示文件。

迁移后的原则：

- Figma Slides 成为唯一主编辑环境。
- 当前仓库保留为内容资产库、视觉参考库、结构参考库。
- 不再追求 PPT 导出与回编兼容。
- 不强行逐像素复刻 web 动效，优先保留叙事、层级、节奏和视觉张力。

## 为什么现在迁

当前仓库已经证明了 web slides 的两个优势：

- 视觉表达更自由
- 页面形式更丰富

但它也带来了明显的维护成本：

- 改稿需要碰代码
- 非前端同事不方便协作
- 导出 PPT 只能截图化，无法获得真正的对象级编辑体验

Figma Slides 正好适合作为这次迁移的中间态：

- 比 PPT 更灵活
- 比代码稿更易编辑
- 适合做演示主稿
- 也更方便后续转为 HTML 或嵌入式展示

## 迁移策略

### 1. 不做一比一翻译

迁移目标不是把当前 web slides 逐像素搬进 Figma。

迁移目标是保留以下 4 件事：

- 论点结构
- 页面节奏
- 视觉语言
- 资产复用关系

以下内容允许降级：

- 复杂背景动态
- 纯代码绘制的轨道、粒子、旋转动画
- 依赖浏览器状态切换的表现

### 2. 以母版和组件重建，而不是逐页手工画

建议先在 Figma Slides 内建立以下基础能力，再开始逐页迁移：

- 颜色变量
- 字体样式
- 页边距系统
- 标题层级
- 组件库
- 母版页

这样可以避免“第一页很好看，后面二十页全靠手调”的问题。

### 3. 复杂页面采用“关键帧表达”

对于 web 中依赖动画和绘制代码的页面，不建议在 Figma 里硬做等价实现。

推荐方式：

- 保留核心构图
- 抽出一帧最能表达观点的状态
- 用 Figma 的层级、透明度、模糊、渐变和简单动效表达气氛
- 把“动态感”转译成“高完成度静态版式”

## 文件结构建议

建议把 Figma 文件分为以下页面：

1. `00 Cover & Theme`
内容：
- 封面实验
- 色板
- 文字层级
- 背景语言

2. `01 Components`
内容：
- 标题组件
- Kicker 组件
- 引导段落组件
- 数据卡组件
- Shot Card 组件
- 双栏对比组件
- 流程步骤组件
- 时间线组件
- Q&A 卡片组件

3. `02 Slides Main`
内容：
- 正式演示页

4. `03 Assets`
内容：
- 截图
- 图表
- 报告页素材
- 备用图形元素

5. `04 Archive`
内容：
- 被替换页面
- 备选方向

## 视觉系统建议

### 颜色

沿用当前 deck 的深色科技感基调，但在 Figma 中收敛成更可维护的变量系统。

推荐变量分层：

- `bg/base`
- `bg/elevated`
- `text/primary`
- `text/secondary`
- `line/subtle`
- `accent/violet`
- `accent/pink`
- `accent/cyan`

### 字体层级

建议在 Figma 中固定 5 套文字样式：

- Display XL：封面/章节页大标题
- Display L：内容页主标题
- Heading M：卡片标题
- Body M：正文
- Meta S：Kicker、编号、辅助标签

### 栅格与间距

建议统一使用 16:9 页面，并建立一致的边距规则。

推荐：

- 左右安全边距：80-96 px
- 上下安全边距：64-80 px
- 常规模块间距：24 / 32 / 40 px
- 大区块间距：56 / 64 px

## 建议先做的母版

### 母版 1：封面页

适用：
- 封面
- 章节开场
- 总结页

结构：
- 左侧大标题
- 辅助说明
- 右侧主视觉或抽象图形

### 母版 2：观点陈述页

适用：
- “AI 是什么”
- “为什么不是更强的搜索”
- “人机分工”

结构：
- 页眉 kicker
- 大标题
- 一段解释
- 1 到 3 个支持点

### 母版 3：双栏对比页

适用：
- 搜索 vs 交互
- 人类 vs AI
- 模糊提问 vs 清晰任务定义

结构：
- 左右两栏
- 中间分隔或箭头
- 每栏有图标、标题、解释

### 母版 4：流程页

适用：
- AI 工作流
- 7 天上手

结构：
- 主标题
- 4 到 7 步的流程条或卡片链

### 母版 5：卡片矩阵页

适用：
- 能力地图
- 国内外入口
- 生产生活场景

结构：
- 标题区
- 2x2、3x2 或横向卡片带

### 母版 6：数据与证据页

适用：
- 为什么是现在
- 研究结论支持页

结构：
- 大标题
- 关键数据
- 图表截图或报告页
- 一句解释

## 资产迁移原则

### 直接复用的资产

来自当前仓库的以下内容可以直接搬进 Figma：

- `assets/screens/`
- `assets/reports/`
- `assets/research/`
- `assets/diagrams/`

### 作为参考重绘的内容

以下内容建议在 Figma 中重建，而不是截图搬运：

- 纯文本和标题排版
- 卡片布局
- 流程图形
- 对比结构
- 编号系统

### 不建议继续保留的内容

- 强依赖 CSS 动画的背景层
- 浏览器交互控件
- 为 web 导航服务的 UI 元素

## 迁移顺序建议

### 第一阶段：建系统

先完成：

- 颜色变量
- 字体样式
- 6 套母版
- 8 到 12 个常用组件

### 第二阶段：迁移高价值页面

优先迁移：

- 封面
- 三个目标
- 搜索 vs 交互
- AI 已进入真实工作流
- 提示词结构
- AI 工作流
- 7 天上手
- 结尾页

这些页面最能决定整套演示的气质和节奏。

### 第三阶段：补足中段信息页

补齐：

- 产品层级
- 国内外入口
- 多方案扩展
- 生产生活场景
- 课后路径

### 第四阶段：清理和统一

最后做：

- 标题长度统一
- 卡片边角与描边统一
- 阴影与模糊统一
- 图片裁切统一
- 页码和章节标签统一

## 关于动画

建议把动画控制在 Figma Slides 容易维护的范围内。

适合保留：

- 淡入
- 位移进入
- 分步出现
- 章节过渡

不建议强做：

- 连续旋转轨道
- 复杂粒子背景
- 高耦合路径动画
- 大量逐元素时间轴动画

## 内容源与责任划分

迁移后建议把仓库的职责改成：

- `src/slides/App.jsx`
  旧版 web slides 参考稿，不再作为主稿维护

- `assets/`
  持续作为素材库维护

- `docs/`
  维护结构说明、迁移清单和脚本说明

Figma 文件的职责改成：

- 页面设计
- 版式调整
- 演示顺序
- 现场分享用主稿

## 完成标准

满足以下条件时，可认为迁移完成：

1. 全部正式页面在 Figma Slides 中可编辑
2. 80% 以上页面使用统一母版或组件搭建
3. 不再依赖 React 项目来改文案和顺序
4. 复杂页面已被成功转译，而不是依赖截图糊过去
5. 最终文件适合非前端同事继续维护

## 结论

这次迁移不是“把一个 web 项目换个地方重画”，而是把当前已经验证过的内容结构与视觉语言，沉淀成一套更适合长期协作的演示系统。

如果迁移顺利，后续的工作方式应该变成：

- 在 Figma Slides 改内容和视觉
- 在仓库里维护素材与说明
- 不再围绕 PPT 兼容做额外工程
